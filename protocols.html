Ethernet

Was ist eigentlich Ethernet?

Unter Ethernet versteht man eine Technologie, die folgende 3 Dinge vereint:

Hardware:

  • Verteiler
  • Netzwerkkarten
  • Kommunikationsmedium, z.B. Nezwerkkabel

Software:

  • Protokolle, die beim Austausch der Daten die Vorgehensweise festlegen, wie vom Rechner empfangene Daten interpretiert werden sollen
  • Zugriffsverfahren, das bestimmt, wie auf den gemeinsamen Übertragungskanal zugegriffen werden darf

Formate:

  • Ethernetframe (Teil des Protokolls)

Es erfolgt somit ein Zusammenspiel aus Hardware, Software und bestimmten Formaten, wenn ich Daten von einem Rechner auf den anderen übertragen möchte.

Das IP-Protokoll

Jedes Gerät im Internet benötigt eine eindeutige Internetadresse, die sogenannte IP-Adresse. Neben Computern und Smartphones brauchen auch smarte Alltagsgegenstände wie der internetfähige Fernseher eine IP-Adresse. Diese Adresse setzt sich aus der Netzadresse und der Hostadresse zusammen. Diese wird als 32-Bit-Zahl codiert und in der Form d1*d2*d3*d4 notiert. Dabei stehen d1*d2*d3*d4 jeweils für die dezimalen Äquivalente der vier Bytes, aus denen die Adresse besteht. Um dem Problem zu entgehen, dass es nicht genügend IP-Adressen geben wird, wurden private IP Adressen definiert. Diese beginnen mit den Nummern „192.68…“. Da aber viele Menschen zu Hause diese Adresse verwenden, muss der Router, der das lokale Netzwerk mit dem Internet verbindet, die private IP-Adresse durch eine gültige Adresse ersetzen. Dies wird als Network Address Translation (NAT) bezeichnet. Zudem wurden die IP-Adressen vergrößert indem das IP Protokoll, das in der Version 4 (IPv4) daherkam, durch eine neue Version 6 (IPv6) ersetzt wurde. In der Version 4 gab es nur 232 mögliche Adressen, arbeitet also mit 32-Bit Adressen, während das neue Internetprotokoll IPv6 mit 2128 Adressen arbeitet. Das IP Protokoll bringt Daten von einem Endgerät (PC, Smartphone) zu einem anderen Endgerät. Es verpackt alle Daten und unterteilt sie in transportable Pakete (Datagramme). Inhalt dieser Pakete ist:

  • Die Adresse des Absenders (Quelladresse)
  • Die Adresse des Empfängers (Zieladresse)
  • Das Feld Time_to_Live (Gültigkeitsdauer, die Daten in Rechnernetzen haben)
  • Das Feld Protocol, das besagt wer beim Empfänger die Daten erhalten soll (z. B Browser oder Email-Programm)

TCP & UPD

TCP

Das TCP (Transmission Control Protocol) ist ein Protokoll, das dazu dient Daten(-pakete) zwischen Entitäten im Netzwerk zu transferieren bzw. zu senden. Es ist die üblichste Form der Kommunikation im World Wide Web, denn überall wo es sich nicht um Data-Streaming handelt, werden in der Regel Daten mit TCP übertragen. Die Besonderheit dieses Protokolls ist das sogenannte Handshake-Prinzip: bei jedem Austausch von Daten(-paketen) wird, findet ein technologische Händeschütteln zwischen Server und Client statt. Dies verläuft ungefähr so:

  1. Client sendet eine Anfrage für die Daten an den Server
  2. Server antwortet mit einer Bestätigung, dass der Transfer stattfinden kann (falls Ports verfügbar)
  3. Client sendet eine Bestätigung, dass der die Antwort vom Server erhalten hat
  4. Verbindungsherstellung und Datentransfer von Server an Client

Ist der ganze Aufwand überhaupt nötig? Klar, das ist das Prinzip von TCP! Wenn es nicht so wäre, könnte es dazu kommen, dass die Datenpakete die der Server sendet gar nicht beim Client ankommen. Somit würden in der Praxis z.B. ein Paar Buchstaben einer WhatsApp Nachricht fehlen oder Teile der CSS-Datei einer Website nicht geladen. Damit es nicht zu solchen Missverständnissen kommt, prüft man sehr genau ob die Verbindung zwischen dem Client und dem Server ausreichend gut ist um einen Datentransfer zu durchführen.
TCP hat jedoch auch ein paar Nachteile, nämlich:

  • Nicht sehr effizient, da bei jedem Datenaustausch 3 Server-Client-Abfragen erfolgen.
  • Einige Webserver-Architekturen (Apache) werden durch TCP angreifbar (Slow Loris Attack)

UDP

UDP (User Datagram Protocol) ist quasi der kleine Bruder von TCP. Der größte Vorteil von UDP dessen Performance, denn die Datenübertragung zwischen Server und Client erfolgt bei UDP ohne die Anfragen und Bestätigungen. Bei UDP werden die Daten einfach regelrecht in eine Richtung geschossen.
UDP garantiert nicht, dass alle Pakete...

  • ...in der richtigen Reihenfolge ankommen
  • ...unverändert/unbeschädigt ankommen
  • ...überhaupt ankommen

Überall wo es sich um Streaming handelt (YouTube, Twitch, Online-Radio, etc…) werden Daten in der Regel mit UDP transferiert. Angenommen man möchte ein 1280x720 Pixel großes Bild per TCP laden. Das hieße für jeden der 921.600 Pixel gäbe es drei Abfragen zwischen dem Client und Server. Drei Abfragen um sicherzustellen, dass alle Pixel richtig und in der richtigen Reihenfolge geladen werden. Spielten jedoch bei fast einer Million Pixel ein, zwei, oder gar 40 Pixel, die verloren gingen eine Rolle? Würde man dies überhaupt bemerken? Nein. Deshalb eigent sich UDP hervorragend für den Austausch großer, hochfrequenter Dateiströme.

HTTP

HTTP (HyperText Transfer Protocol) ist ein Client-Server-Protokoll, dass dem Austausch von Daten im Internet dient. Es bildet die Grundlage des modernen Internets und dessen Datenverkehrs in dem es zum Beispiel dafür sorgt, dass der Browser clientseitig ein HTML-Dokument sowie die zugehörigen Assets in Form von Bildern oder CSS-Files vom Server anfordern kann. In der Regel handelt es sich beim Client um einen Webbrowser, der seine Anfrage (Request) an den Port 80 des Servers schickt, der daraufhin entsprechend antwortet (Response). HTTP wird auch als Application Layer Module bezeichnet, da es auf ein bestehenden Transportprotokoll (z.B. TCP) aufgesetzt werden muss um zu funktionieren.

Requests und Responses bestehen aus einem Header, der die jeweiligen Steuernachrichten enthält, sowie den eigentlichen Daten. Durch das Senden einer Response beendet der Server die vom Client initiierte Verbindung. Zur korrektieren Adressierung der angeforderten Datensätze wird in der Request der sogenannte URL (Uniform Resource Locator) genutzt. Dieser besteht aus:

  • der Protokollangabe http://
  • dem vollständigen Domainnamen des Servers
  • dem Dateinamen & des Dateipfads der gewünschten Datei

Der Server antwortet daraufhin mit einer Response die im Idealfall die angefragte Datei enthält. Auskunft über den Ausführungsstatus gibt dabei der Statuscode der Response. Besonder bekannt ist z.B. der Statuscode 404: ERROR für nichtgefundene Datensätze.

Aus Sicherheitsgründen wird heutzutage jedoch immer mehr Abstand vom klassichen HTTP genommen, da es Daten in unverschlüsseltem Klartext überträgt. Die SSL-verschlüsselte Variante HTTPS löst dieses Problem jedoch.
